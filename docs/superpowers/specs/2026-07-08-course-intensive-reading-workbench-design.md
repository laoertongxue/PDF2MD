# 课程资料精读工作台设计规格

**日期**: 2026-07-08
**项目**: PDF2MD
**状态**: 已确认，待实现计划

## 1. 背景与目标

PDF2MD 的方向从通用“文档解析器”升级为“课程资料精读工作台”。应用仍支持多格式资料导入，但第一阶段核心目标是帮助用户围绕 MBA 重读计划，把课程教材和补充资料转化为高质量章节精读笔记与少量可复用写作卡片。

主场景是：用户为一门课程导入 PDF/Word 主资料，应用转换为 Markdown 并识别章节候选；用户确认章节结构并手动挂载 PPT、Excel、图片等补充资料；应用通过多轮大模型精读生成结构化笔记、两张 Mermaid 图和写作卡片；用户在应用内结构化编辑、单轮重跑，并长期保留同步导出的 Markdown 文件。

## 2. 范围

### 2.1 目标

1. 支持课程级组织，每门课程包含主资料、补充资料、章节、精读笔记和课程卡片池。
2. 支持多格式输入：PDF、Word 作为主资料；PPT、Excel、图片等作为章节补充资料。
3. 主资料先转 Markdown，再自动识别章节候选，必须人工确认后才能进入精读。
4. 每章执行多轮精读，质量优先，不做一次性摘要。
5. 每章最终输出固定栏目：本章概要、核心概念、通俗解释、案例解读、现实应用、延伸思考、写作卡片。
6. 每章必须包含两张可直接预览的 Mermaid 图：知识结构图和应用流程图。
7. 支持 API、Codex CLI、人工任务包三种大模型执行器。
8. 支持结构化块编辑、单轮重跑、运行记录回溯。
9. SQLite 管工作态结构，Markdown 文件作为同步产物长期保留。
10. 课程卡片池自动汇总章节卡片，支持查看、搜索、编辑、收藏、复制。

### 2.2 非目标

- 不做公众号发布。
- 不做文章排期。
- 不做拖卡片生成长文草稿。
- 不做补充资料自动匹配章节。
- 不为 PDF、Word、PPT、Excel、图片分别设计独立完整精读模板。
- 不追求节省 token，第一阶段以精读质量和结果稳定为优先。

## 3. 产品对象

### 3.1 课程

课程是最高层组织单位，例如“战略管理”“营销管理”“组织行为”。课程下包含主资料、补充资料、章节精读结果和课程卡片池。

### 3.2 主资料

主资料第一阶段限定为 PDF 和 Word。它们通常是教材、长文、讲义正文。主资料进入 Markdown 转换、章节识别、人工确认和多轮精读流程。

### 3.3 补充资料

补充资料包括 PPT、Excel、图片等。补充资料不独立进入完整章节精读流程，而是由用户手动挂载到某章，用于增强案例解读、数据理解、图示解释和写作卡片。

### 3.4 章节

章节来自主资料的自动识别候选。用户必须确认章节结构后，章节才能进入多轮精读。确认操作包括改名、合并、拆分、跳过。

### 3.5 精读笔记块

每章最终精读笔记由结构化块组成：

- 本章概要
- 核心概念
- 通俗解释
- 案例解读
- 现实应用
- 延伸思考
- Mermaid 知识结构图
- Mermaid 应用流程图
- 写作卡片入口

### 3.6 写作卡片

每章默认生成 8-12 张高质量卡片。卡片类型包括：

- 选题卡
- 观点卡
- 案例卡
- 类比卡
- 应用卡

卡片正文可以更有公众号写作感，但必须保留来源章节和关联概念。

## 4. 主流程

```text
创建课程
  -> 导入主资料 PDF/Word
  -> 转 Markdown
  -> 自动识别章节候选
  -> 人工确认章节
  -> 手动挂载补充资料
  -> 多轮精读
  -> 结构化编辑和单轮重跑
  -> 同步导出 Markdown
  -> 汇总课程卡片池
```

未确认章节不得进入多轮精读。补充资料第一阶段只支持人工挂载，避免自动误配污染章节理解。

## 5. 多轮精读流水线

每章生成一个章节任务包，包含章节原文 Markdown、补充资料摘要或转换结果、课程/书籍/章节元数据、用户精读规则、输出模板、Mermaid 要求和写作卡片要求。

默认多轮如下：

1. **结构理解轮**：识别本章主题、标题层级、主要问题、教材行文逻辑。
2. **概念模型轮**：提炼核心概念、关键模型、术语关系、教材案例。
3. **通俗解释轮**：生成老师讲课式说明，准确、清楚、生活化、有趣。
4. **应用转化轮**：说明本章知识能解决哪些管理、商业或个人决策问题。
5. **Mermaid 图轮**：生成知识结构图和应用流程图。
6. **写作卡片轮**：生成 8-12 张选题、观点、案例、类比、应用卡片。
7. **自检修订轮**：检查遗漏、格式、Mermaid 可渲染性、栏目完整性和跑题问题，产出最终笔记。

### 5.1 Mermaid 要求

每章必须至少包含两张 Mermaid 图：

1. **知识结构图**：表达本章概念层级、核心模型、关键术语之间的关系。
2. **应用流程图**：表达遇到现实问题时，如何使用本章知识分析和解决。

Markdown 中保留标准 `mermaid` 代码块。应用内必须直接渲染预览，不允许只显示源码。

### 5.2 语言风格

精读正文采用老师讲课型：准确、系统、循序渐进。写作卡片采用公众号写作型：标题感更强、表达更有记忆点、可直接服务贴文和长文素材。

## 6. 大模型执行器

同一份章节任务包可以交给三类执行器：

1. **API 自动执行器**：支持 DeepSeek 和 OpenAI compatible API，用于批量、可重复、自动生成。
2. **Codex CLI 执行器**：本地调用 Codex CLI 跑章节任务包，复用用户现有 Codex/ChatGPT Pro 工作流。
3. **人工任务包执行器**：导出原文、prompt 和输出模板，用户手动投喂模型后再导入结果。

执行器差异不得影响最终数据结构。每轮结果都归档为统一运行记录。

## 7. 重跑与回溯

多轮运行结果默认全部保存在每章 `runs/` 中，应用界面默认折叠。用户可以展开查看每轮输出。

第一阶段支持单轮重跑。重跑某一轮后，依赖它的后续轮次必须标记为需要更新。例如重跑“概念模型轮”后，通俗解释、应用转化、Mermaid、写作卡片和自检修订轮都应标记为过期。

## 8. 存储与文件组织

采用 SQLite + Markdown 双轨。

SQLite 是工作态主存储，负责课程、资料、章节、精读块、Mermaid 图、卡片、收藏状态、模型配置、运行记录和补充资料挂载关系。

Markdown 是同步产物，负责长期可读、可迁移、可备份。应用内编辑后必须同步生成对应 Markdown 文件。

每本主资料输出目录采用每章独立文件夹：

```text
CourseName/
  BookName/
    00-meta/
      book-info.md
      chapter-map.md
    01-第一章/
      source.md
      intensive-note.md
      cards.md
      attachments/
      runs/
    02-第二章/
      source.md
      intensive-note.md
      cards.md
      attachments/
      runs/
```

`source.md` 保存确认后的章节原文。`intensive-note.md` 保存最终精读笔记。`cards.md` 保存本章卡片。`attachments/` 保存手动挂载资料或引用。`runs/` 保存每轮模型输出和任务包。

## 9. 界面

### 9.1 课程页

展示课程列表、当前课程下的主资料和课程卡片池入口。

### 9.2 主资料页

展示文件信息、章节候选列表、章节确认状态和补充资料导入入口。

### 9.3 章节确认页

左侧展示原始章节候选，右侧展示可编辑章节结构。支持改名、合并、拆分、跳过。确认后才允许启动多轮精读。

### 9.4 章节精读页

展示原文 Markdown、补充资料列表、多轮运行状态、最终精读笔记、两张 Mermaid 图、写作卡片、结构化编辑和单轮重跑入口。

Mermaid 图块必须支持源码编辑和实时预览。

### 9.5 课程卡片池

汇总所有章节卡片，支持按类型、章节、关键词筛选，并支持编辑、收藏、复制。

## 10. 验收标准

1. 能创建课程并导入 PDF/Word 主资料。
2. 能把主资料转换为 Markdown 并生成章节候选。
3. 未确认章节不能启动多轮精读。
4. 用户能确认章节结构，并能改名、合并、拆分、跳过章节。
5. 用户能手动把 PPT、Excel、图片等补充资料挂载到指定章节。
6. 每章多轮精读保留中间结果，界面默认折叠。
7. 每章最终笔记包含固定栏目和两张 Mermaid 图。
8. Mermaid 图在应用内直接预览。
9. 每章默认生成 8-12 张写作卡片。
10. 支持 API、Codex CLI、人工任务包三种执行器。
11. 支持单轮重跑，并正确标记后续轮次过期。
12. 应用内结构化编辑后能同步更新 Markdown 文件。
13. 重启应用后能从 SQLite 与 Markdown 恢复课程、章节、笔记和卡片。
14. 课程卡片池支持查看、搜索、编辑、收藏、复制。

## 11. 与现有系统的关系

现有 MarkItDown 适配器、分节器、SQLite repository、FastAPI 服务、React/Tauri 界面可以继续复用。新增工作应聚焦业务层：课程、主资料、章节确认、补充资料挂载、精读任务包、多轮运行记录、结构化笔记块、卡片池和 Markdown 同步。

不要推翻已有解析内核。第一阶段应在现有解析能力之上新增课程精读工作台能力。
